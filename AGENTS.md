# AGENTS.md

## Purpose
このリポジトリは、Chromeの拡張機能を開発するためのものです。
目的は、Home's (https://www.homes.co.jp/search/condition-list/) の検索結果から、不要なものを非表示にしたり、コメントを追加したりすることです
単発の出力よりも、再利用可能で継続運用できる仕組みを優先してください。

## Common core

### 1. Working style
- 明示的に別言語が必要な場合を除き、基本は日本語で書いてください。
- 論理的かつ一貫性を重視してください。
- 以前の判断や提案を変更する場合は、必ず次の3点を明示してください。
  - 何が変わったか
  - なぜ変えたか
  - 解釈がどう変わったか
- 大きな書き換えより、小さくレビューしやすい差分を優先してください。
- 事実を捏造しないでください。検証できないことは、その旨を明確に述べてください。
- commit していないファイルを rm -rf など、ゴミ箱を通さず復元できないような消し方をするときは絶対に事前に確認してください
- シンプルさを第一に：あらゆる変更を可能な限りシンプルにする。影響を受けるコードを最小限に抑える。
- 手抜きは禁物：根本原因を突き止める。その場しのぎの対応はしない。シニア開発者としての基準を守る。
- 最小限の影響：変更は必要な部分のみにとどめるべきです。バグの発生を避けるようにしてください。
- 明示的な要件がない限り、互換性維持のための redirect、fallback、旧 schema 読み、legacy route 温存は入れないでください。互換性を残さないと移行リスクが高いと判断した場合だけ、実装前に理由と影響範囲を短く提案してください。

### 2. Product goal
- 再利用価値のある知識は、可能な限りファイルとして保存してください。
- 長く効くルールと、タスク固有の指示は分けて扱ってください。

### 3. Plan Mode Default
- 3つ以上の手順や設計上の判断を伴うような、些細ではないタスクについては、計画モードに入る
- もし何かがうまくいかなくなったら、すぐに手を止めて計画を立て直してください。無理に続けようとしてはいけません
- ビルドだけでなく、検証手順でもプランモードを使用してください
- 曖昧さを減らすために、事前に詳細な仕様書を作成する

### 4. Self-Improvement Loop
- ユーザーによる修正が行われた後は、docs/lessons.md を以下のパターンで更新してください
- 同じ過ちを繰り返さないように、自分なりのルールを決めておきましょう
- ミス率が低下するまで、これらの教訓を徹底的に反復する
- セッションの冒頭で、該当するプロジェクトの復習を行う

### 5. Verification Before Done
- 動作確認をするまでは、タスクを完了としてマークしてはいけません
- 必要に応じて、メインブランチと変更内容との動作の違い
- 「スタッフエンジニアならこれを承認するだろうか？」と自問してみてください。
- テストを実行し、ログを確認し、正しさを実証する

### 6. Demand Elegance (Balanced)
- 大きな変更を行う際は、一旦立ち止まって「もっと洗練された方法はないか？」と自問してみましょう。
- その修正が「場当たり的」に感じられるなら：「今知っていることをすべて踏まえて、洗練された解決策を実装する」
- 単純で明らかな修正の場合はこの手順を飛ばしてください。不必要に複雑にしないように
- 作品を発表する前に、まず自分自身でその内容を見直してみましょう

### 7. Autonomous Bug Fixing
- バグ報告を受けたら、ただ修正してください。手取り足取りのサポートを求めないでください
- ログ、エラー、失敗したテストを特定し、それらを解決する
- ユーザーによるコンテキスト切り替えは一切不要です
- 指示されなくても、失敗しているCIテストを修正してください

### 8. Task Management
- まず計画を立てる：チェック可能な項目を tasks/todo.md に書き出す
- `docs/todo.md` は commit 単位の TODO とし、各項目に「なぜその変更を入れるのか」が短く分かる意図を残してください。
- `docs/todo_vX.Y.md` は大きい Phase 改修の正本とし、Phase ごとの順序と意図を残してください。
- 計画の確認：実施を開始する前に確認を行う
- 進捗状況の確認：完了した項目にチェックを入れていく
- 変更点の説明：各ステップごとの概要
- ドキュメントの結果: tasks/todo.mdにレビューセクションを追加する
- 授業の記録：修正後、docs/lessons.mdを更新する

### 9. Engineering principles
- 強い理由がない限り、既存のアーキテクチャとパターンに従ってください。
- 明確な命名、小さなモジュール、低結合、明示的なインターフェースを優先してください。
- 早すぎる抽象化は避けてください。
- 新しいパターンを導入する場合は、現在の実装よりなぜ良いのかを説明してください。

### 10. Branch discipline
- 開発は `main` で直接行わないでください。
- 作業前に現在の branch を確認し、`main` にいる場合は必ず作業用 branch を切ってから始めてください。 
- すでに `main` に未マージで、かつ今回の作業内容に合う branch がある場合は、その branch で作業してください。
- 開発 branch の標準形は `codex/<topic>-<action>-<what>[-<detail>]` とし、英小文字とハイフンだけを使ってください。
- 該当する作業中 branch がない場合は、内容に合った名前の branch を新しく切ってください。未マージの継続 branch だけは再利用してよく、`main` 追随のための rebase は許可します。
- merged / closed branch は再利用しないでください。同名が必要なときは、まず `detail` を具体化し、それでも足りない場合のみ `-v2`, `-v3` を使ってください。
- commit は細かめに行い、機能改修またはバグ修正ごとにレビューしやすく追いやすい単位へ分けてください。
- `main` への反映は、作業 branch 上の commit を review 可能な単位でまとめたうえで、merge 経由で行ってください。
- マージは明示的な指示がない限り `--no-ff` でマージしてください。
- user が「この branch / commit を `main` に merge したい」「`A` を親にして `B` を merge した形にしたい」と履歴形状を指定した場合は、内容同等の cherry-pick / revert / 再適用で代用しないでください。まず「欲しいのは同じ変更内容か、同じ親子関係を持つ履歴か」を切り分け、履歴自体が要件なら base commit と merge 親を commit SHA で確認してから操作してください。

### 11. Test-first development
- 振る舞い変更がある場合は、可能であれば先に失敗するテストを書くか、既存テストを更新してください。
- その後、テストを通すための最小変更を実装してください。
- リファクタリングは、関連テストが通ったあとに行ってください。
- 影響範囲の検証なしに、作業を完了扱いにしないでください。

### 12. Quality gates
作業完了とみなす前に、次を行ってください。
- まず最小の関連テストを実行する
- 設定されていれば lint / format を実行する
- 設定されていれば typecheck / build を実行する
- diff を見直し、不要な変更がないか確認する
- リスク、制約、未対応事項、次の対応候補を要約する

### 13. [DOCS] UX / design principles
- 落ち着いていて、モダンで、情報密度は高いが読みやすい UI を優先してください。
- 装飾の派手さより、明確さ、余白、情報階層、素早いスキャン性を重視してください。
- 既存のコンポーネントやトークンがある場合は再利用してください。
- 挙動は予測可能で、アクセシブルであることを重視してください。

### 14. [DOCS] Document versioning
- `docs/` 配下の正本 doc は、ファイル名に `_vX.Y` サフィックスを付けてください。
- doc の一番大きい見出しには、ファイル名と同じ version を明記してください。
- 軽微な更新は `v1.0 -> v1.1 -> v1.2 ...` のように minor を進めてください。
- 大幅改定のときだけ major を上げて `v2.0` のようにしてください。
- version を上げたら、その系統の古い doc は削除し、最新 version のファイルだけを残してください。
- ただし `docs/todo_vX.Y.md` は例外です。大きい改修の milestone 履歴として複数 version を残してよく、Phase が 1 から始まるような改修だけ version を上げてください。
- version を上げたら、AGENTS、README、関連 docs、Skills の参照先も同じ change で最新 version へ更新してください。

### 15. Research and content principles
- 可能であれば一次情報や公式情報を優先してください。
- 事実、解釈、仮説を分けて扱ってください。
- 記事やポストを生成するときは、出典追跡可能性を保ってください。
- リサーチ、DMログ、ユーザー反応から得た知識は、チャット出力だけで終わらせず、再利用可能なファイルに保存してください。

### 16. [SKILL] Skills boundary
- 長い手順書をこのファイルに直接書かないでください。
- ルールが長くなる場合は、docs/ または Skills に移し、このファイルから参照してください。

## 20. Codex-specific

### 21. Execution preferences
- 実装の前に、関連ファイルと既存パターンを確認してください。
- コード編集時は、最小差分を優先してください。
- 不確かな場合は、推測で進めず、選択肢とトレードオフを示してください。

### 22. [DOCS] Validation loop
コード変更時は、次の順で進めてください。
1. 最小の関連テストを特定する
2. まずそのテストを実行する
3. 通るための最小変更を実装する
4. ローカルで成功したあとにだけ、より広い検証を行う
5. 何を検証し、何をまだ検証していないかを要約する

### 23. Skills usage
- タスクが反復可能なワークフローに一致する場合は、即興で進める前に該当 Skill を確認してください。
- 毎回ゼロから考え直すより、repository 内の Skills と references を優先してください。
