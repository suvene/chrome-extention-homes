# Lessons

- 共有グループの所属を変える処理では、変更前に見えていた state を先に確保してから group を更新する。理由: unlink の直後に state を再解決すると、共有先だけに残っていたメモを current listing から見失うことがあるため。
- `chrome.storage.sync` を段階的に外すときでも、移行コードと運用コードを分けて確認する。理由: 「移行のために読む」実装と「運用で書く」実装が混ざると、local-only 化の完了条件を誤解しやすいため。
- `紐づき件数` や `紐づけ一覧` は、自分自身を含めずに数える。理由: 自分自身を 1 件として見せると、実際の shared 相手がいないのに「解除できるはず」と誤解しやすいため。
- 自動候補の基準と、ユーザーが明示的に指定できる基準は分ける。理由: fingerprint だけでは拾えない重複掲載があり、detail URL のような手動キーを別入口で受けられると運用が安定するため。
- 行単位の更新が主目的なら、一覧の右側に行アクションを置く。理由: checkbox + 全体更新よりも、`リンク` / `解除` の方が対象と結果の対応が読み取りやすいため。
- 複数条件を組み合わせる候補判定では、集合ごとの独立一致ではなく、どの基準データに対する `AND` 条件かを先に固定する。理由: 名前と住所のような条件を別々の既知候補へまたいで満たす誤判定を防ぐため。
- 一覧内の表示位置指定は、候補種別ごとの例外を推測せず、`紐づき中` `リンク` `もしかして` の各行でどこに出すかを明示的に整理してから実装する。理由: badge 行と詳細行のように表示段が複数ある UI では、部分的な例外解釈がずれると確認往復が増えるため。
- 「保存されるが表示されない」不具合を疑うときは、保存処理より先に行種別ごとの描画条件を確認する。理由: 一覧 UI では `if` 条件 1 つで特定の行だけ表示対象から外れていることがあり、保存経路の調査より早く原因に届くため。
- サイト別の `もしかして` 判定では、表示名や住所の表記揺れを `SITE_CONFIGS` とは別の比較専用 helper へ閉じ込める。理由: `athome` のように「物件名 + 階数」や都道府県省略が混ざるサイトでは、表示や fingerprint を変えずに候補判定だけ補正したい場面があるため。
- サイト側が card 全体をクリック遷移にしている場合は、panel の mount 位置だけでなく panel 境界でイベント伝播を止める。理由: 見た目上 panel が入っていても、親要素の click handler が生きていると入力・編集操作が詳細遷移に吸われて UX が壊れるため。
- サジェスト候補に「確認」と「実行」の両方があるときは、物件名リンクで確認し、実行は明示ボタンに分ける。理由: 候補カード全体クリックで即リンクさせると、内容確認のつもりの操作が状態変更になりやすいため。
